This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository does

This is the corrections management system for the [Cologne Sanskrit Lexicon](https://www.sanskrit-lexicon.uni-koeln.de/) (CSL). It:
- Collects user-submitted corrections via a PHP correction form (`app/correction_form.php`) on the Cologne server
- Stores cumulative responses in `app/correction_response/cfr.tsv` (8-column TSV: timestamp, dict, Lnum, headword, old, new, comment, status)
- Distributes corrections per dictionary into `dictionaries/<dict>/` subdirectories
- Maintains `xxx_printchange.txt` files that document intentional deviations from the printed editions (as opposed to digitization errors)

The sister repository `csl-orig` holds the actual dictionary source files (`csl-orig/v02/<dict>/<dict>.txt`), which is what corrections are ultimately applied to.

## Key scripts

**`sh redo_cfr.sh`** â€” Main daily workflow. Runs `cfr_adj.py` to regenerate `correctionform.txt` and all `dictionaries/xxx/xxx_correctionform.txt` files from `cfr.tsv`.

**`sh post_github_issues.sh`** â€” Uploads pending corrections as GitHub issues to `sanskrit-lexicon/csl-orig`. Reads `last_cfr_line.txt` as a counter to track which lines have already been uploaded.

**`sh update_user_corrections.sh <dict>`** â€” After manually editing `csl-orig/v02/<dict>/<dict>.txt`, rebuilds the local dictionary installation, validates XML, and commits/pushes to `csl-orig`.

**`python3 cfr_adj.py app/correction_response/cfr.tsv correctionform.txt`** â€” Parses `cfr.tsv` and writes per-dictionary correctionform files. Recognises known dict codes via the `knowndicts` list inside the script; add new codes there when a new dictionary is introduced.

**`python3 issues/issue87/issue87.py <dict_code> [change_file_path]`** â€” Tags occurrences in `csl-orig` dictionary text with correction markers. Run from the `issues/issue87/` directory. Looks up `../../dictionaries/<dict>/<dict>_printchange.txt` by default, or accepts an explicit path. Writes output to `temp_<dict>.txt`.
- Uses `{{old->new}}` tags for `printchange` files
- Uses `[[old->new]]` tags for other change files
- Requires `csl-orig` to be checked out as a sibling of this repo (`../../../csl-orig/v02/<dict>/<dict>.txt`)

## File formats

**`cfr.tsv`** â€” 8 tab-separated columns: `timestamp | dict | Lnum | headword | old | new | comment | status`. Lines without `:` in the last field are "pending". Lines with `:` in the last field are already resolved.

**`xxx_printchange.txt`** â€” Free-text file with dated sections. Change lines must match: `<Lcode> : <headword> : <old> : <new> : <comment>` (colon-delimited, first character is a digit). Lines starting with `Source:`, `Ref:`, or `L code :` are header lines and are skipped by issue87.py.

**`xxx_correctionform.txt`** â€” Generated by `cfr_adj.py`. Each case block has `Case N:`, `old =`, `new =`, `comment =`, `status =` lines separated by `--------`.

**`daily/YYYYMMDD/`** â€” Snapshots created during daily correction runs; contain `cfr-YYYYMMDD.tsv`, `correctionform-YYYYMMDD.txt`, and a `dictionaries/` sub-tree.

## Dictionary codes

Cologne codes used throughout (directory names under `dictionaries/`): `acc ae ap ap90 ben bhs bop bor bur cae ccs fri gra gst ieg inm krm lan lrv mci md mw mw72 mwe pd pe pgn pui pw pwg pwkvn sch shs skd stc vcp vei wil yat`.

## Workflow (typical daily corrections cycle)

1. New user submissions arrive in `app/correction_response/cfr.tsv`
2. Run `sh redo_cfr.sh` to regenerate per-dictionary correctionform files
3. Review pending items, edit `csl-orig/v02/<dict>/<dict>.txt` manually
4. Mark resolved lines in `cfr.tsv` by appending `: <date> corrected` to the status field
5. Run `sh post_github_issues.sh` to open GitHub issues for remaining pending items
6. Run `sh update_user_corrections.sh <dict>` for each corrected dictionary
7. Commit and push this repository

## GitHub Issue Conventions

All issues follow the Sanskrit Lexicon taxonomy shared across the org.

### Milestones

| # | Title | Types |
|---|---|---|
| 1 | Dictionary to Book | `link-target`, `link-splitting` |
| 2 | Digitization Quality | `text-correction`, `encoding`, `bug`, `scan-quality` |
| 3 | Structured Data | `markup`, `question` |
| 4 | Major Enhancements | `content-enhancement` |

### Type labels â€” color `#0075ca`

| Label | When to use |
|---|---|
| `link-target` | Building click-throughs from `<ls>` abbreviations to scanned PDF pages |
| `link-splitting` | Splitting combined `SOURCE N,N` refs into individual per-page links |
| `markup` | Normalising XML tag content (`<ls>`, `<lex>`, `<ab>`, etc.) |
| `text-correction` | Corrections to dictionary text (definitions, headwords) |
| `content-enhancement` | New material, display upgrades, structural additions beyond correction |
| `encoding` | SLP1/AS/IAST transcoding, character rendering, hyphen/dash normalisation |
| `scan-quality` | Replacing blurry, skewed, or missing scan pages |
| `bug` | Broken links, XML errors, broken scripts or download files |
| `question` | Scholarly or editorial questions requiring research before any code change |

### Severity labels

| Label | Color | When to use |
|---|---|---|
| `minor` | `#e4e669` | Targeted, self-contained fix |
| `medium` | `#fbca04` | Standard unit of work â€” one index, a batch of corrections |
| `hard` | `#d93f0b` | Large effort spanning many sources, files, or dictionaries |

## Data format

### Markup tags in `csl-orig` dictionary source files

| Tag | Role | Example |
|---|---|---|
| `<L>NNNN` | Entry begin, with print line number | `<L>12345` |
| `<LEND>` | Entry end | |
| `<k1>headword` | Primary headword in SLP1 | `<k1>rAma` |
| `<k2>variant` | Secondary spelling or transliteration | `<k2>rAma` |
| `<e>N` | Homonym number | `<e>1` |
| `<lex>code` | Lexical category (gender, part of speech) | `<lex>m.` |
| `<ls>source` | Literary source citation | `<ls>Rv. 1.22.16` |
| `<ab>tag` | Italicised abbreviation | `<ab>m.</ab>` |
| `{#text#}` | Sanskrit text in SLP1 | `{#rAmaH#}` |
| `{%text%}` | Italicised display text | `{%abc%}` |

- `<L>8687` â€” print line number used for `updateByLine.py` change files

- `{#uttaraNga#}` â€” Devanagari/IAST Sanskrit text
- `<lex>a.</lex>` â€” adjective marker
- `<ab>m.</ab>` â€” gender abbreviation (masculine)
- `<LEND>` â€” entry close
