# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository does

This is the corrections management system for the [Cologne Sanskrit Lexicon](https://www.sanskrit-lexicon.uni-koeln.de/) (CSL). It:
- Collects user-submitted corrections via a PHP correction form (`app/correction_form.php`) on the Cologne server
- Stores cumulative responses in `app/correction_response/cfr.tsv` (8-column TSV: timestamp, dict, Lnum, headword, old, new, comment, status)
- Distributes corrections per dictionary into `dictionaries/<dict>/` subdirectories
- Maintains `xxx_printchange.txt` files that document intentional deviations from the printed editions (as opposed to digitization errors)

The sister repository `csl-orig` holds the actual dictionary source files (`csl-orig/v02/<dict>/<dict>.txt`), which is what corrections are ultimately applied to.

## Key scripts

**`sh redo_cfr.sh`** — Main daily workflow. Runs `cfr_adj.py` to regenerate `correctionform.txt` and all `dictionaries/xxx/xxx_correctionform.txt` files from `cfr.tsv`.

**`sh post_github_issues.sh`** — Uploads pending corrections as GitHub issues to `sanskrit-lexicon/csl-orig`. Reads `last_cfr_line.txt` as a counter to track which lines have already been uploaded.

**`sh update_user_corrections.sh <dict>`** — After manually editing `csl-orig/v02/<dict>/<dict>.txt`, rebuilds the local dictionary installation, validates XML, and commits/pushes to `csl-orig`.

**`python3 cfr_adj.py app/correction_response/cfr.tsv correctionform.txt`** — Parses `cfr.tsv` and writes per-dictionary correctionform files. Recognises known dict codes via the `knowndicts` list inside the script; add new codes there when a new dictionary is introduced.

**`python3 issues/issue87/issue87.py <dict_code> [change_file_path]`** — Tags occurrences in `csl-orig` dictionary text with correction markers. Run from the `issues/issue87/` directory. Looks up `../../dictionaries/<dict>/<dict>_printchange.txt` by default, or accepts an explicit path. Writes output to `temp_<dict>.txt`.
- Uses `{{old->new}}` tags for `printchange` files
- Uses `[[old->new]]` tags for other change files
- Requires `csl-orig` to be checked out as a sibling of this repo (`../../../csl-orig/v02/<dict>/<dict>.txt`)

## File formats

**`cfr.tsv`** — 8 tab-separated columns: `timestamp | dict | Lnum | headword | old | new | comment | status`. Lines without `:` in the last field are "pending". Lines with `:` in the last field are already resolved.

**`xxx_printchange.txt`** — Free-text file with dated sections. Change lines must match: `<Lcode> : <headword> : <old> : <new> : <comment>` (colon-delimited, first character is a digit). Lines starting with `Source:`, `Ref:`, or `L code :` are header lines and are skipped by issue87.py.

**`xxx_correctionform.txt`** — Generated by `cfr_adj.py`. Each case block has `Case N:`, `old =`, `new =`, `comment =`, `status =` lines separated by `--------`.

**`daily/YYYYMMDD/`** — Snapshots created during daily correction runs; contain `cfr-YYYYMMDD.tsv`, `correctionform-YYYYMMDD.txt`, and a `dictionaries/` sub-tree.

## Dictionary codes

Cologne codes used throughout (directory names under `dictionaries/`): `acc ae ap ap90 ben bhs bop bor bur cae ccs fri gra gst ieg inm krm lan lrv mci md mw mw72 mwe pd pe pgn pui pw pwg pwkvn sch shs skd stc vcp vei wil yat`.

## Workflow (typical daily corrections cycle)

1. New user submissions arrive in `app/correction_response/cfr.tsv`
2. Run `sh redo_cfr.sh` to regenerate per-dictionary correctionform files
3. Review pending items, edit `csl-orig/v02/<dict>/<dict>.txt` manually
4. Mark resolved lines in `cfr.tsv` by appending `: <date> corrected` to the status field
5. Run `sh post_github_issues.sh` to open GitHub issues for remaining pending items
6. Run `sh update_user_corrections.sh <dict>` for each corrected dictionary
7. Commit and push this repository
